# 개발 워크플로우 규칙 (always-on)

> 이 파일은 `CLAUDE.md`가 `@import`로 끌어와 **매 세션 항상** 적용된다.
> 이 파일은 특정 기능에만 적용되는 것이 아니라 프로젝트의 모든 개발 작업에 공통으로 적용된다.

## 절대 하지 않는다 (금지)

* ❌ 스펙과 계획서 없이 소스 코드 작성 또는 수정
* ❌ 사용자의 명시적인 승인 전에 구현 시작
* ❌ `docs/specs/`에 없는 기능을 임의로 추가
* ❌ `main` 브랜치에 직접 커밋하거나 Push
* ❌ 사용자 승인 없이 구현 계획이나 범위 변경
* ❌ 애매한 요구사항을 임의로 해석
* ❌ PR 리뷰 없이 변경사항을 `main`에 병합
* ❌ 모든 변경사항을 하나의 거대한 커밋으로 처리
* ❌ `wip`, `more changes`처럼 변경 내용을 알 수 없는 커밋 메시지 사용

## 파이프라인

1. **Spec** ('docs/specs/<feature-slug>.md')
구현할 기능의 **무엇을 만들 것인지와 왜 필요한지**를 작성한다.
계획서를 작성하기 전에 사용자와 합의해야 한다.
* [ ] 문제 정의
* [ ] 기능을 구현하는 목적
* [ ] 기능 요구사항
* [ ] 입력값과 데이터 구조
* [ ] Validation 규칙
* [ ] 상태값과 상태 전환 조건
* [ ] 인수 조건(Acceptance Criteria)
* [ ] 구현 범위
* [ ] 명시적인 제외 범위(Out of Scope)

2. **Plan** ('docs/plans/<feature-slug>.md')
스펙이 확정되면 구현 방법을 작성한다.
소스 코드는 수정하지 않는다.
* [ ] 단계별 구현 방법
* [ ] 추가하거나 수정할 파일과 모듈
* [ ] 데이터베이스 변경사항
* [ ] 구현 순서
* [ ] 단계별 검증 방법
* [ ] 예상되는 위험 요소
* [ ] 해결되지 않은 질문
* [ ] 로컬 테스트 방법

3. **사용자 승인**
작성한 계획서를 사용자에게 제시하고 명시적인 승인을 받는다.
승인 여부가 명확하지 않으면 사용자에게 다시 확인한다.
승인 전에는 소스 파일을 작성하거나 수정하지 않는다.
계획이 변경되면 변경된 계획을 다시 제시하고 재승인을 받는다.
가능하면 Plan Mode를 사용하여 계획 제시와 구현 단계를 명확히 분리한다.
승인으로 인정할 수 있는 답변의 예시: 승인합니다, 좋아요, 진행해 주세요.

4. **Branch**
계획이 승인되면 독립적인 기능 브랜치를 생성한다.
브랜치 형식: feature/<feature-slug>
브랜치의 `<feature-slug>`는 스펙 및 계획서 파일명과 일치해야 한다.
`main` 브랜치에는 직접 커밋하거나 Push하지 않는다.
GitHub의 브랜치 보호 규칙에 따라 `main` 반영은 반드시 PR을 통해 진행한다.

5. **구현**
승인된 계획에 따라 `feature/<feature-slug>` 브랜치에서 구현한다.
구현 도중 스펙 범위 밖의 작업이 필요해지면 즉시 구현을 멈춘다.
이때 사용자에게 상황을 알리고 Spec과 Plan을 수정한 뒤 다시 승인을 받아야 한다.
* 승인된 스펙과 계획 범위 안에서만 구현한다.
* 계획서에 작성된 구현 순서와 대상 파일을 따른다.
* 작업에 맞는 도메인 스킬을 적용한다.
* 계획서를 실행할 때는 `executing-plans` 스킬을 사용한다.
* 기존 기능에 영향을 미치는 경우 회귀 테스트를 수행한다.
* 커밋은 파일 단위가 아닌 의미 있는 작업 단위로 나눈다.

6. **로컬 검증**
구현이 완료되면 PR을 생성하기 전에 애플리케이션을 로컬에서 실행하고 결과를 확인한다.
* [ ] 애플리케이션이 정상적으로 빌드되는지 확인
* [ ] 자동화 테스트 실행
* [ ] 브라우저에서 실제 기능 확인
* [ ] 데이터가 PostgreSQL에 정상 저장되는지 확인
* [ ] Validation과 오류 메시지 확인
* [ ] 기존 기능의 회귀 여부 확인
* [ ] 스펙의 인수 조건과 결과 대조

7. **PPull Request 및 Merge**
기능 브랜치를 원격 저장소에 Push하고 `main` 브랜치를 대상으로 PR을 생성한다.
PR 설명에는 다음 내용을 포함한다.
* 무엇을 변경했는지
* 왜 변경했는지
* 어떻게 검증했는지
* UI 변경 시 결과 화면 또는 스크린샷

## 예외: 계획 없이 허용되는 작업

다음과 같은 읽기 전용 작업은 Spec, Plan, 승인 또는 기능 브랜치 없이 수행할 수 있다.
단, 탐색 중 문제를 발견했더라도 사용자 승인 없이 바로 수정하지 않는다. 수정이 필요하면 별도의 Spec과 Plan 절차를 시작한다.
* 프로젝트 파일과 폴더 읽기
* 코드 검색
* Git 상태와 로그 확인
* 기존 구현 구조 분석
* 애플리케이션을 실행하여 현재 동작 관찰
* 테스트를 실행하여 현재 상태 확인
* 오류 로그 확인
* 변경 없이 원인 분석
* 사용자에게 현재 프로젝트 상태 설명

## 문서 체계

| 경로                     | 용도                       |
| ---------------------- | ------------------------ |
| `docs/specs/`          | 무엇을 왜 구현하는지 정의하는 요구사항 문서 |
| `docs/specs/README.md` | 스펙 문서 작성 양식과 안내          |
| `docs/plans/`          | 승인받을 단계별 구현 계획서          |
| `docs/plans/README.md` | 계획서 작성 양식과 안내            |
| `.claude/rules/`       | 모든 작업에 항상 적용되는 개발 규칙     |
| `.claude/skills/`      | 특정 작업에서 호출하는 구현 및 검토 절차  |
