# 코드 리뷰 표준 (code-review-standard)

## 언제 사용하는가
* 사용자가 **"이 변경 검토해줘"**라고 요청할 때
* 사용자가 **"코드 리뷰해줘"**라고 요청할 때
* 사용자가 **"PR 봐줘"**라고 요청할 때
* `executing-plans`로 구현이 완료된 뒤 셀프 리뷰할 때
* Pull Request를 생성하거나 병합하기 전
* 기능 완료 여부를 최종 판단하기 전
* 기존 변경사항이 스펙과 계획에 맞는지 확인할 때

## 리뷰 대상
* 대상 스펙
* 승인된 구현 계획
* 변경된 소스 코드
* 추가되거나 수정된 테스트
* Git diff
* 커밋 기록
* Pull Request 설명
* 테스트 및 빌드 결과


# 리뷰 체크리스트

## 1. 스코프
변경이 승인된 범위 안에 있는지 확인한다.
* [ ] 대상 스펙의 요구사항만 구현했다.
* [ ] 승인된 계획에 정의된 파일과 모듈 중심으로 변경되었다.
* [ ] 관계없는 기존 코드를 임의로 리팩터링하지 않았다.
* [ ] 스펙에 없는 기능이 추가되지 않았다.
* [ ] 계획에 없던 의존성이 추가되지 않았다.
* [ ] 불필요한 설정 변경이 포함되지 않았다.
* [ ] 자동 생성 파일이나 로컬 환경 파일이 커밋되지 않았다.

## 2. 정확성
구현이 요구사항대로 동작하는지 확인한다.
* [ ] 정상 흐름이 요구사항대로 동작한다.
* [ ] 잘못된 입력이 적절히 거부된다.
* [ ] 빈 문자열, null, 빈 목록을 처리한다.
* [ ] 잘못된 상태 전이를 막는다.
* [ ] 중복 요청이나 중복 데이터 상황을 처리한다.
* [ ] 존재하지 않는 리소스 접근을 처리한다.
* [ ] 경계값이 올바르게 처리된다.
* [ ] 예외가 적절한 계층에서 처리된다.
* [ ] 오류 응답이 프로젝트 규격과 일치한다.
* [ ] 기존 기능의 동작이 깨지지 않았다.
* [ ] 데이터 저장과 조회 결과가 일관적이다.
* [ ] 트랜잭션이 필요한 작업에 적절히 적용되었다.

## 3. 보안
사용자 입력, 권한, 출력, 민감정보를 확인한다.

### 입력 검증
* [ ] 사용자 입력에 길이, 형식, 범위 검증이 있다.
* [ ] 서버 측 검증이 적용되어 있다.
* [ ] 클라이언트 검증만 신뢰하지 않는다.
* [ ] 파일 업로드가 있다면 확장자·MIME·크기 검증이 있다.
* [ ] 숫자나 ID를 직접 입력받을 때 접근 권한을 다시 검증한다.

### 인증 및 권한
* [ ] 인증이 필요한 엔드포인트가 보호되어 있다.
* [ ] 관리자 권한이 필요한 기능에 권한 검사가 있다.
* [ ] 요청자의 리소스 소유권을 확인한다.
* [ ] URL이나 ID를 바꾸는 것만으로 다른 사용자의 데이터에 접근할 수 없다.
* [ ] Service 계층에서도 필요한 권한 검증이 유지된다.

### XSS
* [ ] Thymeleaf에서 `th:utext`를 불필요하게 사용하지 않는다.
* [ ] 동적 HTML 출력이 기본 이스케이프를 우회하지 않는다.
* [ ] Vanilla SSR에서는 동적 출력이 `escapeHtml`을 거친다.
* [ ] 사용자 입력을 HTML 문자열에 직접 삽입하지 않는다.
* [ ] URL, 속성값, 스크립트 문맥에 맞는 이스케이프를 적용한다.

### 민감정보
* [ ] 비밀번호, 토큰, API 키가 코드에 하드코딩되지 않았다.
* [ ] 비밀값은 `.env` 또는 설정 파일을 통해 주입된다.
* [ ] `.env`가 Git에 커밋되지 않는다.
* [ ] 로그에 비밀번호, 토큰, 개인정보가 출력되지 않는다.
* [ ] API 응답에 내부 예외나 스택 트레이스가 노출되지 않는다.
* [ ] 데이터베이스 접속 정보가 저장소에 노출되지 않는다.

### 하드코딩
* [ ] 비밀키가 하드코딩되지 않았다.
* [ ] 사용자 ID나 관리자 ID가 고정값으로 들어가 있지 않다.
* [ ] 토큰 만료시간이 코드 곳곳에 중복 하드코딩되지 않았다.
* [ ] 환경에 따라 달라지는 값이 설정으로 분리되어 있다.

## 4. 컨벤션
프로젝트의 기술별 표준을 따르는지 확인한다.
* [ ] 클래스와 메서드 이름이 역할을 명확히 표현한다.
* [ ] 하나의 클래스가 지나치게 많은 책임을 가지지 않는다.
* [ ] 중복 코드가 불필요하게 생기지 않았다.
* [ ] 지나친 추상화가 없다.
* [ ] 사용하지 않는 코드나 import가 남아 있지 않다.
* [ ] 기존 코드 스타일과 포맷을 유지한다.
* [ ] 관련 없는 주석이나 docstring을 삭제하지 않았다.

### Spring Backend
* [ ] Controller는 요청·응답 처리에 집중한다.
* [ ] 비즈니스 로직은 Service 계층에 있다.
* [ ] Repository가 비즈니스 판단을 담당하지 않는다.
* [ ] Entity를 API 응답으로 직접 노출하지 않는다.
* [ ] DTO가 요청과 응답 용도에 맞게 분리되어 있다.
* [ ] 예외 처리가 `GlobalExceptionHandler` 규칙을 따른다.
* [ ] 트랜잭션 범위가 적절하다.
* [ ] 의존성 주입 방식이 프로젝트 표준을 따른다.

### Thymeleaf UI
* [ ] `th:text`를 우선 사용한다.
* [ ] `th:utext` 사용에 명확한 이유가 있다.
* [ ] 폼 입력과 서버 검증 오류가 연결되어 있다.
* [ ] URL은 `@{...}` 형식을 사용한다.
* [ ] 화면에 비즈니스 로직이 과도하게 들어가지 않는다.
* [ ] 공통 레이아웃과 Fragment를 재사용한다.

### JPA / PostgreSQL
* [ ] 연관관계의 주인이 올바르게 설정되어 있다.
* [ ] 불필요한 양방향 연관관계가 없다.
* [ ] N+1 문제가 발생하지 않는지 확인했다.
* [ ] 유니크 제약과 인덱스가 요구사항에 맞게 설정되어 있다.
* [ ] `equals`와 `hashCode`가 Entity 특성에 맞다.
* [ ] 감사 컬럼이 프로젝트 규칙에 맞게 적용되었다.
* [ ] DB 제약조건과 애플리케이션 검증이 서로 모순되지 않는다.
* [ ] 대량 조회 시 페이징이 적용되어 있다.

### Frontend SSR
* [ ] HTML Fragment와 페이지 역할이 분리되어 있다.
* [ ] 동적 출력이 `escapeHtml`을 거친다.
* [ ] 공통 레이아웃을 중복 작성하지 않는다.
* [ ] dev/prod 경로 분기가 올바르다.
* [ ] 정적 리소스 경로가 빌드 결과와 일치한다.
* [ ] 브라우저에서 깨진 링크나 누락된 리소스가 없다.

## 5. 테스트
새 로직을 검증하는 테스트와 기존 기능의 회귀 여부를 확인한다.

### 자동 테스트
* [ ] 새 기능에 대응하는 테스트가 있다.
* [ ] 실패 흐름에 대한 테스트가 있다.
* [ ] 엣지 케이스 테스트가 있다.
* [ ] 기존 테스트가 모두 통과한다.
* [ ] 테스트가 구현 세부사항보다 동작을 검증한다.
* [ ] 테스트가 서로 독립적으로 실행된다.
* [ ] 테스트 데이터가 다른 테스트에 영향을 주지 않는다.
* [ ] 테스트를 통과시키기 위해 검증을 약화하지 않았다.
* [ ] 실패한 테스트를 삭제하거나 비활성화하지 않았다.

### 수동 검증
* [ ] 애플리케이션을 실제로 실행했다.
* [ ] 브라우저에서 대상 화면을 확인했다.
* [ ] 정상 입력 흐름을 확인했다.
* [ ] 오류 입력 흐름을 확인했다.
* [ ] 기존 핵심 화면과 기능을 확인했다.
* [ ] API를 직접 호출해 응답을 확인했다.
* [ ] DB 저장 결과가 요구사항과 일치하는지 확인했다.


# 심각도 등급
## 🔴 Blocker
완료 또는 머지 전에 반드시 수정해야 하는 문제다.
* 요구사항과 다르게 동작함
* 데이터 손실 가능성
* 인증·권한 누락
* XSS, SQL Injection 등 보안 취약점
* 스펙 위반
* 테스트 실패
* 빌드 실패
* 민감정보 노출
* 기존 핵심 기능 회귀
* 승인되지 않은 범위의 변경

## 🟡 Should
고치는 것이 좋지만 반드시 머지를 막지는 않는 문제다.
* 컨벤션 불일치
* 테스트 커버리지 부족
* 중복 코드
* 이해하기 어려운 네이밍
* 불필요하게 큰 메서드
* 개선 가능한 예외 메시지
* 리뷰하기 어려운 커밋 구조

## 🟢 Nit
기능이나 안정성에 영향을 거의 주지 않는 사소한 의견이다.
* 네이밍 선호
* 줄바꿈
* 주석 표현
* import 순서
* 메시지 문구
* 작은 스타일 차이

## 절차
1. 위 체크리스트로 변경된 파일을 점검하고 각 발견 사항에 등급을 매긴다.
2. 🔴 Blocker는 완료 처리 전에 반드시 해소한다.
3. 최종 확인은 사용자가 한다 — AI 리뷰는 1차 필터일 뿐, 최종 승인을 대체하지 않는다.
4. 무엇을·왜·어떻게 검증했는지 사용자에게 짧게 보고한다.

## 하지 않는 것
* 스펙을 확인하지 않고 코드 스타일만 리뷰
* 계획 범위를 벗어난 변경을 문제없이 승인
* Blocker가 남아 있는데 완료 또는 승인 처리
* 테스트 실패를 무시하고 병합 가능하다고 판단
* 검증하지 않은 내용을 통과했다고 표현
* 개인 취향을 Blocker로 분류
* 근거 없이 대규모 리팩터링 요구
* 스펙에 없는 기능 구현을 리뷰 조건으로 요구
* 기존 주석이나 docstring 삭제를 불필요하게 요구
* 사용자의 승인 없이 코드를 직접 병합
* 사용자의 승인 없이 브랜치를 삭제
* 보안 문제를 단순 Nit으로 낮춰 표현
* 중요 문제를 모호한 표현으로 숨김
