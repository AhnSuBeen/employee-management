# 코드 리뷰 표준 (code-review-standard)

이 프로젝트는 git/PR을 쓰지 않는 로컬 과제이므로, "리뷰"는 변경된 파일을 직접 점검하는
셀프 리뷰를 뜻한다.

## 체크리스트
- **스코프** — 변경이 대상 스펙(`docs/specs/`)·계획(`docs/plans/`) 범위 안에 있는가? 스펙에 없는 기능이 슬쩍 들어오지 않았는가?
- **정확성** — 요구사항대로 동작하는가? 엣지 케이스(빈 값, 잘못된 상태 전이 등)를 처리하는가?
- **보안** — 사용자 입력이 검증되는가? Thymeleaf에서 `th:utext` 오남용 등으로 XSS 위험이 없는가? 민감정보가 로그/응답에 노출되지 않는가?
- **컨벤션** — `spring-backend`/`thymeleaf-ui`/`jpa-postgres` 스킬의 표준(계층 분리, 네이밍, 감사 컬럼 등)을 따르는가?
- **테스트** — 새 로직에 대응하는 테스트가 있는가(있는 경우)? 최소한 수동 검증(브라우저 동작 확인)이 되어 있는가?

## 심각도 등급
- 🔴 **Blocker** — 완료 처리 전 반드시 고쳐야 함 (정확성 오류, 보안 문제, 스펙 위반 등).
- 🟡 **Should** — 고치는 게 좋음이지만 완료를 막지는 않음 (컨벤션 불일치, 개선 여지).
- 🟢 **Nit** — 사소한 의견 (네이밍, 스타일 등).

## 절차
1. 위 체크리스트로 변경된 파일을 점검하고 각 발견 사항에 등급을 매긴다.
2. 🔴 Blocker는 완료 처리 전에 반드시 해소한다.
3. 최종 확인은 사용자가 한다 — AI 리뷰는 1차 필터일 뿐, 최종 승인을 대체하지 않는다.
4. 무엇을·왜·어떻게 검증했는지 사용자에게 짧게 보고한다.

## 하지 않는 것
- 스펙 범위를 벗어난 변경을 별다른 논의 없이 완료 처리
- Blocker가 남은 채로 완료 보고
