# 계획 실행 (executing-plans)

## 전제 조건
- `docs/plans/<slug>.md`가 존재하고 사용자가 명시적으로 승인했다.
- 승인되지 않았다면 코드를 건드리지 않고 승인부터 요청한다.

이 프로젝트는 git을 사용하지 않는 로컬 과제다 — 브랜치 생성이나 커밋 단계는 없다.
계획서의 작업 단위를 그대로 순서대로 구현한다.

## 절차
1. **계획을 체크포인트로 사용** — 계획서의 단계 목록을 순서대로 따라간다. 각 단계를 완료할 때마다 진행 상황을 남긴다.
2. **기존 코드 존중** — 이번 작업과 무관한 기존 주석·docstring·코드는 삭제하거나 임의로 리팩터링하지 않는다.
3. **스코프 이탈 감지** — 계획에 없던 요구사항이나 엣지 케이스를 발견하면 멈추고 사용자에게 알린다. 계획서를 갱신하고 재승인받은 뒤 계속한다. 계획에 없는 기능을 임의로 추가하지 않는다.
4. **회귀 확인** — 관련 기존 기능이 깨지지 않았는지 확인한다(빌드/테스트/수동 확인).
5. **검증 게이트** — 구현이 끝나면 실제로 실행해 브라우저 등에서 동작을 확인하고, 원 스펙의 수용기준과 대조한다. 결과를 walkthrough로 정리해 사용자에게 보고한다.
6. **셀프 리뷰** — 병합/제출 전에 `code-review-standard` 체크리스트로 변경 사항을 스스로 점검한다.

## 하지 않는 것
- 사용자 승인 없이 계획 변경
- 스펙/계획에 없는 기능 임의 추가
- 변경 대상이 아닌 기존 주석·docstring 삭제

- 리뷰: `code-review-standard`
