# 계획 실행 (executing-plans)

## 전제 조건
* `docs/plans/<filename>.md` 형태의 구현 계획서가 존재한다.
* 해당 계획서를 사용자가 명시적으로 승인했다.
* 계획서에 대상 스펙과 작업 범위가 명시되어 있다.
* 계획서에 사용할 Git 브랜치명이 정의되어 있다.
* 현재 Git 작업 디렉터리 상태를 확인할 수 있다.

## Git 기본 원칙
* 기능별 작업 브랜치를 사용한다.
* 기본 브랜치에서 직접 구현하지 않는다.
* 계획서에 명시된 브랜치명을 우선 사용한다.
* 각 Task 또는 의미 있는 작업 단위마다 커밋한다.
* 기존의 관련 없는 변경사항을 임의로 커밋하지 않는다.
* 커밋 전에 테스트와 변경 범위를 확인한다.
* 모든 작업 완료 후 병합, Pull Request 또는 브랜치 유지 방법을 사용자에게 제시한다.

## 절차
1. **계획서 불러오기**
승인된 구현 계획서를 읽은 후, 다음 항목을 확인한다.
* 구현 목표
* 대상 스펙
* 아키텍처와 접근 방식
* 기술 스택
* 작업 브랜치명
* Task 목록
* 각 Task에서 생성·수정할 파일
* 테스트 방법
* 검증 명령어
* 커밋 단위
* 리스크
* 미해결 질문

2. **계획 비판적 검토**
코드를 수정하기 전에 계획을 실제 프로젝트와 다시 대조한다.
* 계획에 빠진 핵심 단계가 없는가?
* 계획의 파일 경로가 실제 프로젝트 구조와 일치하는가?
* 이미 존재하는 기능을 다시 만들도록 되어 있지 않은가?
* 구현할 수 없는 단계가 포함되어 있지 않은가?
* 테스트 방법이 현재 프로젝트에서 실행 가능한가?
* 명령어가 현재 운영체제와 빌드 환경에 맞는가?
* 요구사항이나 수용 기준이 불명확하지 않은가?
* 계획서와 현재 코드 사이에 충돌이 없는가?
* 현재 브랜치에 다른 작업자의 미완성 변경사항이 있지 않은가?

3. **Git 상태 확인**
먼저 현재 저장소 상태를 확인한다.
* 현재 위치가 올바른 Git 저장소인가?
* 작업 디렉터리에 커밋되지 않은 변경사항이 있는가?
* 현재 브랜치가 기본 브랜치인가?
* 계획서에 명시된 작업 브랜치가 이미 존재하는가?
* 기존 변경사항이 이번 작업과 관련 있는가?
관련 없는 수정 사항이 있으면 삭제하거나 덮어쓰지 않는다.
기존 작업과 충돌 가능성이 있으면 작업을 중단하고 사용자에게 알린다.

4. **기존 코드 존중**
이번 작업과 직접 관계없는 기존 코드를 임의로 변경하지 않는다.
특히 다음 항목을 삭제하거나 수정하지 않는다.
* 기존 주석
* docstring
* 로그
* 포맷
* 변수명
* 메서드 구조
* 파일 구성
* 다른 기능의 리팩터링
* 사용자가 작성 중인 코드

5. **작업 단위마다 커밋**
Task가 완료되고 검증에 성공하면 변경사항을 확인한다.
* 이번 Task와 관련된 파일만 변경되었는가?
* 디버깅 코드가 남아 있지 않은가?
* 비밀키나 환경변수 파일이 포함되지 않았는가?
* 테스트 파일이 포함되었는가?
* 자동 생성 파일이 불필요하게 포함되지 않았는가?

6. **스코프 이탈 감지**
구현 중 다음 상황이 발생하면 현재 Task를 중단한다.
* 계획에 없는 요구사항을 발견함
* 새로운 엣지 케이스가 기존 설계를 바꿔야 함
* 데이터베이스 구조를 추가로 변경해야 함
* 새로운 외부 라이브러리가 필요함
* 다른 모듈을 광범위하게 수정해야 함
* 계획서의 접근 방식으로 수용 기준을 만족할 수 없음
* 예상보다 큰 리팩터링이 필요함

7. **회귀 확인**
구현 완료 시 관련 기존 기능이 깨지지 않았는지 확인한다.
* 관련 단위 테스트
* 통합 테스트
* 전체 테스트
* 애플리케이션 빌드
* 서버 실행
* 브라우저 수동 확인
* API 요청 확인
* 데이터베이스 동작 확인
* 기존 주요 사용자 흐름 확인

8. **검증 게이트**
실제 애플리케이션을 실행해 동작을 확인한다.
* 서버가 정상적으로 실행되는가?
* 브라우저에서 대상 화면이 표시되는가?
* API 요청과 응답이 정상적인가?
* 정상 입력이 예상대로 처리되는가?
* 잘못된 입력이 적절히 거부되는가?
* 오류 메시지가 요구사항과 일치하는가?
* 기존 기능이 깨지지 않았는가?
* 원래 스펙의 모든 수용 기준이 충족되는가?

9. **셀프 리뷰**
병합/제출 전에 `code-review-standard` 체크리스트로 변경 사항을 스스로 점검한다.

## 하지 않는 것
* 승인되지 않은 계획 실행
* 사용자 승인 없이 계획 변경
* 스펙이나 계획에 없는 기능 추가
* 기본 브랜치에서 직접 기능 구현
* 관련 없는 기존 코드 리팩터링
* 기존 주석이나 docstring 삭제
* 테스트를 실행하지 않고 완료 선언
* 실패한 테스트를 삭제하거나 비활성화하여 통과 처리
* 다른 사람의 미완성 변경사항 덮어쓰기
* 관계없는 파일을 함께 커밋
* 사용자의 동의 없이 강제 Push
* 사용자의 동의 없이 브랜치 삭제
* 사용자 승인 없이 병합
* 오류를 숨기고 다음 Task로 진행
* 반복 실패 상황에서 근거 없이 추측하여 수정
