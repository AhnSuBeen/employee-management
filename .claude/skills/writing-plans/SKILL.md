# 계획서 작성 (writing-plans)

## 언제 사용하는가
- 사용자가 "계획서부터 써줘"라고 명시적으로 요청할 때
- `docs/specs/`의 스펙이 확정된 뒤, 코드를 쓰기 전 단계
- 기존 기능을 변경/확장할 때도 동일하게 적용한다 — 사소해 보여도 생략하지 않는다

## 절차
1. **기획 자료 재확인**
- 대상 스펙(`docs/specs/<slug>.md`)을 다시 읽은 후, 다음 항목을 확인한다.
* 요구사항은 무엇인가?
* 수용 기준(Acceptance Criteria)은 무엇인가?
* 구현 범위에 포함되는 것은 무엇인가?
* 구현 범위 밖의 항목은 무엇인가?
* 빠진 요구사항은 없는가?
* 모호하거나 서로 충돌하는 요구사항은 없는가?
- 빠지거나 모호한 부분이 있다면 **추측하지 않는다.**
- 사용자에게 질문하여 요구사항을 명확하게 한 뒤 계획을 작성한다.

2. **기존 코드 대조**
- 관련 기존 코드를 읽은 후, 다음 항목을 확인한다.
* 이미 존재하는 엔티티가 있는가?
* 이미 존재하는 Controller 또는 API가 있는가?
* 재사용할 수 있는 Service 로직이 있는가?
* 기존 Repository 쿼리를 활용할 수 있는가?
* 기존 화면이나 템플릿을 재사용할 수 있는가?
* 현재 프로젝트의 아키텍처 및 코딩 스타일은 무엇인가?
* 새로운 작업이 기존 구조와 어떻게 연결되는가?
- 이미 존재하는 기능이나 구조를 다시 만들지 않는다.

3. **계획서 작성**
- `docs/plans/YYYY-MM-DD-<주제>.md` 파일로 작성한다.
- 계획서에는 최소한 다음 내용을 포함해야 한다.
| 항목         | 설명                                |
| ---------- | --------------------------------- |
| 대상 스펙 링크   | 어떤 `docs/specs/*.md` 파일을 기반으로 하는지 |
| 목표         | 무엇을 구현하는지 한 문장으로 설명               |
| 아키텍처       | 구현 접근 방식을 2~3문장으로 설명              |
| 기술 스택      | 사용하는 핵심 기술 및 라이브러리                |
| 접근 방식      | 어떤 방식으로 구현할 것인지 설명                |
| 대안 및 선택 이유 | 필요한 경우 다른 방법과 비교하고 선택 이유 설명       |
| 단계별 작업 목록  | 실제 구현 순서                          |
| 대상 파일 및 모듈 | 각 Task에서 생성·수정·테스트할 파일            |
| 테스트 전략     | 어떤 테스트를 먼저 작성할 것인지                |
| 검증 방법      | 어떤 명령어로 구현 성공 여부를 확인할 것인지         |
| 브랜치명       | `feature/<slug>` 형식               |
| 리스크        | 예상되는 기술적 또는 구현상의 위험               |
| 미해결 질문     | 아직 결정되지 않은 사항                     |

4. **승인 요청**
- 계획서 초안을 작성한 후 사용자에게 제시하고 **명시적인 승인**을 받는다.
- 승인 전에는 다음 작업을 수행하지 않는다.
* 소스 코드 수정
* 설정 파일 수정
* 데이터베이스 스키마 변경
* 새로운 의존성 추가
* 실제 기능 구현
* 스펙에 없는 기능 추가
- 승인 전에는 Plan Mode를 유지한다.

5. **승인 후에만** `executing-plans` 스킬로 넘어간다.

## 하지 않는 것
* 사용자 승인 전에 코드 수정
* 사용자 승인 전에 설정 파일 수정
* 계획서 없이 바로 구현 시작
* 스펙에 없는 기능을 계획에 몰래 추가
* 기존 코드를 확인하지 않고 새로운 구조 생성
* 모호한 요구사항을 임의로 추측
* 테스트 및 검증 단계를 생략
* 구현 중 계획과 다른 방향으로 조용히 스코프 확대

## 참고
- 계획이 진행 중 바뀌면 계획서를 갱신하고 다시 승인받는다 — 조용히 스코프를 넓히지 않는다.
