# 계획서 작성 (writing-plans)

## 언제 사용하는가
- 사용자가 "계획서부터 써줘"라고 명시적으로 요청할 때
- `docs/specs/`의 스펙이 확정된 뒤, 코드를 쓰기 전 단계
- 기존 기능을 변경/확장할 때도 동일하게 적용한다 — 사소해 보여도 생략하지 않는다

## 절차
1. **기획 자료 재확인** — 대상 스펙(`docs/specs/<slug>.md`)을 다시 읽는다. 요구사항·수용기준·범위 외 항목을 명확히 한다. 빠지거나 모호한 부분이 있으면 추측하지 말고 사용자에게 질문한다.
2. **기존 코드 대조** — 이미 있는 엔티티·컨트롤러·템플릿·설정을 실제로 읽고, 새 작업이 기존 구조와 어떻게 맞물리는지 확인한다. 있는 것을 다시 만들지 않는다.
3. **계획서 작성** — `docs/plans/YYYY-MM-DD-<주제>.md` 파일로 작성한다(오늘 날짜 접두어). 최소 다음을 포함한다.
   - 대상 스펙 링크
   - 접근 방식 및 (필요한 경우) 대안과 선택 이유
   - 단계별 작업 목록 — 각 단계마다 건드릴 파일/모듈 명시
   - 사용할 브랜치명 (`feature/<slug>`)
   - 리스크·미해결 질문
4. **승인 요청** — 계획서 초안을 사용자에게 제시하고 명시적 승인을 받는다. Plan Mode를 사용해 승인 전에는 소스 코드를 건드리지 않는다.
5. **승인 후에만** `executing-plans` 스킬로 넘어간다.

## 하지 않는 것
- 승인 전에 코드/설정 파일 수정
- 스펙에 없는 기능을 계획에 슬쩍 포함
- 계획서 없이 구현으로 바로 진입 (아무리 작아 보여도)

## 참고
- 계획이 진행 중 바뀌면 계획서를 갱신하고 다시 승인받는다 — 조용히 스코프를 넓히지 않는다.
