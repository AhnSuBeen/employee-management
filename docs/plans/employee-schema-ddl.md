# 직원 테이블 스키마를 schema.sql DDL로 관리 — 구현 계획

스펙: `docs/specs/employee-schema-ddl.md`

## 목표
`employee` 테이블 스키마를 Hibernate 자동 생성(`ddl-auto: update`) 대신 명시적인
`schema.sql` DDL 파일로 관리하도록 전환한다.

## 아키텍처
Spring Boot는 데이터소스 초기화 스크립트(`schema.sql`)를 EntityManagerFactory 생성
이전에 실행한다(`spring.jpa.defer-datasource-initialization` 기본값 `false`). 즉
`schema.sql`이 먼저 실행되어 테이블을 준비한 뒤, Hibernate가 그 스키마를 엔티티와
대조(`ddl-auto: validate`)하는 순서가 된다. PostgreSQL은 임베디드 DB가 아니므로
`spring.sql.init.mode: always`를 명시해야 `schema.sql`이 실행된다. DDL은
`CREATE TABLE IF NOT EXISTS`로 작성해, 테이블이 이미 있는 기존 로컬 DB에서는
아무 변경 없이 통과하고, 테이블이 없는 새 환경에서는 새로 생성한다.

## 기술 스택
- Spring Boot SQL 초기화(`spring.sql.init.*`) — 별도 마이그레이션 도구(Flyway/Liquibase)
  없이 순수 SQL 스크립트만 사용.
- PostgreSQL 17 (Docker) / 로컬 실행 중인 PostgreSQL 인스턴스.

## 접근 방식
1. 현재 `Employee`/`BaseEntity` 엔티티의 `@Column` 매핑을 그대로 DDL로 옮긴다 — 새로운
   컬럼이나 제약을 추가하지 않는다(스펙에서 확정한 범위 그대로).
2. `ddl-auto: update`는 `validate`로 바꾼다 — 이제 스키마의 출처는 `schema.sql`이고,
   Hibernate는 엔티티와 실제 스키마가 일치하는지 검증만 한다.
3. `spring.sql.init.mode: always`를 추가해 PostgreSQL에서도 `schema.sql`이 실행되게 한다.
   `defer-datasource-initialization`은 기본값(`false`)을 그대로 둔다 — 스크립트가 먼저
   실행되어야 `validate` 검증이 통과하기 때문.

### 대안 및 선택 이유
- **Flyway/Liquibase 도입**: 버전 관리형 마이그레이션이 정석이지만, 스펙에서 이번 범위는
  단순 `schema.sql`로 확정했으므로 채택하지 않는다(향후 필요 시 별도 스펙으로 진행).
- **`DROP TABLE` 후 재생성**: 기존 데이터를 지우게 되어 스펙의 "기존 테이블/데이터 유지"
  요구사항에 위배되므로 채택하지 않는다.

## 구현 단계
1. **`src/main/resources/schema.sql` 작성**
   - `CREATE TABLE IF NOT EXISTS employee (...)` — 컬럼: `id`(`BIGINT GENERATED BY
     DEFAULT AS IDENTITY PRIMARY KEY`), `name`/`department`/`position`/`email`/`phone`
     (`VARCHAR(255) NOT NULL`), `hire_date`(`DATE NOT NULL`), `status`(`VARCHAR(255)
     NOT NULL`), `created_at`/`updated_at`(`TIMESTAMP NOT NULL`).
   - 타입·nullable은 `Employee.java`/`BaseEntity.java`의 현재 `@Column` 매핑과 정확히
     일치시킨다(불일치 시 `ddl-auto: validate`가 기동을 실패시킴).

2. **`src/main/resources/application.yaml` 수정**
   - `spring.jpa.hibernate.ddl-auto`: `update` → `validate`.
   - `spring.sql.init.mode: always` 추가.
   - 주석 처리된 상단 블록(1~10번 줄)은 이번 변경과 무관하므로 그대로 둔다.

3. **엔티티 코드 변경 없음**
   - `Employee.java`, `BaseEntity.java`는 스펙상 컬럼 구성 변경이 없으므로 그대로 둔다.
   - DDL이 기존 매핑과 어긋나지 않는지만 재확인한다.

## 대상 파일 및 모듈
- `src/main/resources/schema.sql` (신규 생성)
- `src/main/resources/application.yaml` (수정)

## 테스트 전략
- 별도의 단위/통합 테스트는 추가하지 않는다 — 스키마 초기화 설정 변경이라 자동화 테스트
  대상이 아니며, 검증은 아래 "검증 방법"의 실제 기동 확인으로 대체한다.
- 기존 `EmployeeManagementApplicationTests`(context load)가 변경 후에도 통과하는지 확인한다.

## 검증 방법
1. `./gradlew.bat test` — 기존 테스트가 모두 통과하는지 확인(특히 `contextLoads`).
2. **기존 데이터가 있는 로컬 DB**에서 `./gradlew.bat bootRun` 실행 → 정상 기동 로그 확인,
   테이블이 재생성되지 않고 기존 데이터가 그대로 남아 있는지 브라우저(`/employees` 목록)로
   확인.
3. **테이블이 없는 빈 환경**에서 한 번은 별도로 확인한다(기존 로컬 데이터에 영향을 주지
   않는 임시 DB/컨테이너 사용) → `schema.sql`이 테이블을 생성하고 애플리케이션이 정상
   기동하는지 확인. 이 단계는 실 데이터에 영향이 없는 별도 환경에서만 수행한다.
4. 기존 기능(목록/등록/상세/수정/상태변경)이 스키마 전환 후에도 동일하게 동작하는지
   브라우저로 회귀 확인.

## 브랜치명
`feature/employee-schema-ddl`

## 리스크
- 엔티티의 `@Column` 매핑과 손으로 작성한 DDL의 타입/제약이 미세하게 어긋나면
  `ddl-auto: validate` 단계에서 애플리케이션이 기동 실패한다 — Hibernate가 실제로
  생성했던 것과 동일한 타입(`VARCHAR(255)`, `TIMESTAMP` 등)을 그대로 옮겨 위험을
  최소화한다.
- `spring.sql.init.mode: always`는 매 기동마다 스크립트를 실행하므로, 향후 `schema.sql`에
  `CREATE TABLE IF NOT EXISTS` 이외의 구문(예: `ALTER TABLE`)을 추가할 경우 재실행 시
  오류가 날 수 있다 — 이번 범위에서는 `CREATE TABLE IF NOT EXISTS` 하나만 두어 문제되지
  않는다.
