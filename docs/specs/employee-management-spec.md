# 직원관리 시스템 (Employee Management) — 요구사항 정의서

상태: 데이터 모델(ERD)·상태값 코드·필드 검증 규칙·기능 범위(F-05 포함, 페이지네이션 보류)
확정됨. 계획서(`docs/plans/`) 작성 단계로 진행 가능 — 남은 항목은 아래 "열린 질문" 참고.
관련 문서: `.claude/rules/development-workflow.md`, `docs/plans/README.md`,
`docs/specs/employee-list-search-filter.md`(F-05 상세 스펙)

## 배경 / 목적
직원(emp) 정보를 조회·등록·수정하고 재직 상태를 관리하는 사내 웹 애플리케이션을 만든다.
목적은 화려한 기능이 아니라, 아래 요구사항 범위 안에서 플랜-퍼스트 워크플로우
(`.claude/rules/development-workflow.md`)를 그대로 따르는 것이다. 기획(본 문서)은 이미
정의되어 있으므로, 이후 단계는 계획(`docs/plans/`) 수립 → 승인 → 구현 순서로 진행한다.

## 스택 (고정)
- Java 21, Gradle, Spring Boot 3.5
- Spring MVC + Thymeleaf (서버 사이드 렌더링, SPA/별도 프론트엔드 프레임워크 없음)
- Spring Data JPA + PostgreSQL
- Spring Validation, Lombok

배포는 하지 않는다 — 로컬 실행(`./gradlew.bat bootRun`)으로 동작을 확인한다.

## 기능 요구사항

### F-01 직원 목록 조회
직원 목록을 표로 보여준다. 주요 필드와 재직 상태를 함께 표시하고, 상태는 배지(badge)
형태로 표시한다.

### F-02 직원 등록
입력 폼으로 새 직원을 등록한다. 정의된 검증 규칙을 통과해야 저장된다.

### F-03 직원 상세 / 수정
직원 한 명의 상세 정보를 조회하고 수정할 수 있다.

### F-04 상태 변경
재직 → 휴직 → 복직(재직) → 퇴직 상태 전이를 지원한다. 상태 변경 시 목록의 배지가
즉시 반영되어야 한다.

### F-05 검색 · 필터 (확정 — 범위 포함으로 변경)
이름 · 부서 · 직급으로 목록을 필터링하는 기능. 처음에는 이번 구현 범위에서 제외하기로
했었으나, 이후 요청에 따라 범위에 포함하는 것으로 결정을 뒤집었다. 상세 요구사항은
`docs/specs/employee-list-search-filter.md`에 별도로 정의한다.

## 공통 상태 정의
모든 구현에서 동일하게 맞추는 유일한 공통 규칙이다. (확정)

| 상태 | 코드값 | 의미 |
|---|---|---|
| 재직 | `ACTIVE` | 정상 근무 중 |
| 휴직 | `ON_LEAVE` | 복직 가능한 일시 중단 상태 |
| 퇴직 | `RESIGNED` | 종료 상태 (더 이상 전이 없음) |

전이 규칙: `ACTIVE` ⇄ `ON_LEAVE` → `RESIGNED`.

## API 스펙 (개요)
REST 엔드포인트 구성은 아래를 따른다 (경로/메서드는 고정, 요청·응답 바디 상세는
계획 단계에서 확정한다).

| Method | Path | 설명 |
|---|---|---|
| GET | `/employees` | 목록 조회 |
| POST | `/employees` | 등록 |
| GET | `/employees/{id}` | 상세 조회 |
| PATCH | `/employees/{id}` | 수정 |
| PATCH | `/employees/{id}/status` | 상태 변경 |

Thymeleaf 화면은 이 API 또는 동등한 MVC 컨트롤러 흐름 위에서 목록 / 등록·수정 폼 /
상세 페이지로 구성한다.

## 데이터 모델 / ERD (확정)
`Employee` 엔티티는 아래 컬럼으로 구성한다:

| 필드 | 설명 | 검증 규칙 |
|---|---|---|
| `id` | PK | 자동 생성 |
| `name` | 이름 | 필수 |
| `department` | 부서 | 필수 |
| `position` | 직급 | 필수 |
| `email` | 이메일 | 필수, 이메일 형식 |
| `phone` | 전화번호 | 필수, 전화번호 형식 |
| `hire_date` | 입사일 | 필수 |
| `status` | 상태 (`ACTIVE`/`ON_LEAVE`/`RESIGNED`) | 필수, 등록 시 기본값 `ACTIVE` |
| `created_at`/`updated_at` | 공통 감사 컬럼 | `jpa-postgres` 스킬 참고, 앱에서 자동 채움 |

검증 범위는 필수 여부 + `email`/`phone` 형식 검증까지로 한정한다 (유니크 제약, 길이 제한
등 추가 규칙은 두지 않는다). 전화번호 형식의 정확한 패턴(예: `010-0000-0000` 하이픈
포함 여부)은 계획 단계에서 구체화한다.

## 범위 제외 (Out of scope)
- 배포/운영 환경 구성 (로컬 실행만 대상)
- 인증/인가 (요구사항에 명시되지 않음)
- 목록 페이지네이션 (나중으로 보류하기로 확정)
- 인사 발령 이력 등 추가 엔티티 (진행 여부 미확정 — 필요 시 별도 요청)
